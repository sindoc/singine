Summary

Four command families were implemented and documented for managing the full lifecycle of a web property inside singine: deployment, validation, build, and security. All commands are registered in singine/command.py and available in the installed CLI.

CommandPurposeCollibra
singine www Deploy pipeline: git → build → Dropbox → rsync/scp WWW-DEPLOY-001
singine vww Read-only validation: TLS, HTTP headers, asset audit VWW-AUDIT-001
singine wingine Build engine: cortex/python · maven-war WING-BUILD-001
singine wsec TLS certs · ed25519 SSH keys · IDP JWT tokens WSEC-IDP-001

Deploy pipeline

Source (git) │ │ singine www pull ▼ Local working copy │ │ singine wingine build ▼ Built output (html/ or target/webapp) │ ├─── singine www sync --method dropbox │ → ~/Dropbox/www/<site>/ │ └─── singine www sync --method rsync → deploy@<host>:/var/www/... (via singine IDP SSH key) Security surface ├─── singine wsec keys — ed25519, registered as urn:singine:deploy:<site> ├─── singine wsec cert — TLS check / certbot renewal ├─── singine wsec token — JWT HS256, iss: urn:singine:idp, TTL 3600s └─── singine vww audit — full read-only security report

Decisions & findings

markupware.com TLS certificate WSEC-CERT-001
singine vww cert --site markupware.com reveals ERR_TLS_CERT_ALTNAME_INVALID: the server returns a Let's Encrypt certificate issued for yesplay-online.com. This is a virtual host misconfiguration — the wrong default vhost is selected before SNI resolves. Required SANs are entirely absent.
Fix: singine wsec cert --site markupware.com --fix-san --method certbot
IDP trust chain WSEC-IDP-001
SSH deploy keys are ed25519, stored in ~/.singine/keys/, registered as urn:singine:deploy:<site> in singine.jks via singine.sec.trust. Deploy tokens are HS256 JWTs signed by singine.auth.token (iss: urn:singine:idp, TTL 3600 s).
Build backends WING-BUILD-001
markupware.com uses the python-cortex backend (XML source → HTML via xsltproc + DocBook XSL, catalogued via cortex/catalog.xml rewriting dead remote URLs to local archives). lutino.io uses the maven-war backend. Both serve locally via Python's built-in HTTP server.
vww HTTP security scan VWW-AUDIT-001
Checks: HTTPS redirect · HSTS · CSP · X-Content-Type-Options · X-Frame-Options · Referrer-Policy · Permissions-Policy · server header exposure. All read-only. Complements wsec cert with application-layer posture.
